# Track Rename (Design)

**Date:** 2026-05-03
**Scope:** small feature — let the user override a track's display name from the UI, with a side effect of correcting the lyrics-tab metadata
**Server:** `127.0.0.1:8765`

## Goal

Today, a track's user-visible "name" comes from two unrelated sources:

- **Topbar title pill, Tools menu, browser tab** — derived from the on-disk filename via `deriveTitle(...)` in `webui/static/js/ui/topbar.js`. For slug-style files (`charlie_puth_attention.mp3`) this produces "Charlie Puth Attention" — wrong (no separator), good enough for human display.
- **Lyrics tab header** — `meta.artist` / `meta.title` from `cache/<slug>/lyrics/meta.json`, populated by `identify_track()` on first fetch. For slug-style files these end up as `("", "charlie_puth_attention")` — broken, and the contenteditable affordance to fix it inline is unusable because keyboard shortcuts on the canvas (space, digits) capture the keys before the contenteditable sees them.

This feature adds a single user-authored "display name" that overrides the filename-derived title across all surfaces and, on save, splits into artist/title to repair the lyrics meta. The slug — the filesystem identity — is **not** changed.

## Non-goals

- Renaming the slug, the on-disk MP3, the cache directory, or the URL parameter (much bigger feature, separate spec if ever needed).
- Bulk rename across the library.
- Auto-refetch of LRCLIB lyrics after a rename — the existing "Refetch from LRCLIB" item in the lyrics ▾ menu already covers this, and renames don't always mean the user wants new lyrics fetched.
- Replacing the lyrics-tab contenteditable with a separate inline editor. The contenteditable becomes effectively read-only display; this rename modal is the canonical edit path.

## Architecture

A new server endpoint, a new persistence file, and a small UI modal:

| Layer | What | Where |
|---|---|---|
| **Persistence** | `cache/<slug>/user_meta.json` — `{"display_name": "..."}` | New file, added to the `PRESERVE` set in `_clear_cache_dir` so reanalyze doesn't destroy it |
| **Server merge** | `tracks.get_summary(slug)` reads `user_meta.json` and merges `display_name` into the returned `track` dict | `webui/webui/tracks.py` |
| **API** | `PATCH /api/tracks/{slug}` — body `{"display_name": "..."}` | `webui/webui/server.py` |
| **Side effect** | On PATCH, also update `cache/<slug>/lyrics/meta.json` artist/title via the smart split | Server-side, in the PATCH handler |
| **UI** | ✎ pencil button right of the title pill; click opens a modal; Save → PATCH | `webui/static/js/ui/topbar.js` + new `webui/static/js/ui/rename-modal.js` |
| **Surfaces** | Topbar title, Tools menu, track picker, browser tab `<title>`, lyrics tab header all read `track.display_name` with `deriveTitle(track.file)` as fallback | Touch points across `topbar.js`, `menus.js`, `track-picker.js`, `main.js`, `lyrics-tab.js` |

## Persistence

### Why `user_meta.json` and not `summary.json`

`summary.json` is regenerated by every `python -m analyze` run — anything stored there is destroyed on reanalyze. `webui/webui/analyze_runner.py:48-49` already documents the preserved set as `chat.json` and `lyrics/`. We extend that set with `user_meta.json`. Using a dedicated file keeps user-authored state separated from pipeline-generated state, and the file is small enough that the proliferation cost is negligible.

### File shape

```json
{
  "display_name": "Charlie Puth - Attention"
}
```

That's it for now. Future user-editable metadata can land in the same file (BPM override, key override, etc.); the schema is intentionally an open object.

### Reanalyze interaction

Add `"user_meta.json"` to `PRESERVE` in `_clear_cache_dir`. Existing tests cover the preserve behavior for `chat.json`/`lyrics/`; we'll extend them.

## Server endpoint — `PATCH /api/tracks/{slug}`

### Request

```http
PATCH /api/tracks/charlie_puth_attention
Content-Type: application/json

{"display_name": "Charlie Puth - Attention"}
```

### Validation

- `display_name` is required, must be a string
- Trimmed length in `[1, 200]`
- Reject if it contains any of `\`, `/`, `\n`, `\r`, `\0` (defense — could end up in `document.title`, in cache filenames if ever extended)
- 4xx with a clear `detail` on rejection

### Smart split for lyrics meta

```python
name = body["display_name"].strip()
if " - " in name:
    artist, _, title = name.partition(" - ")
    artist, title = artist.strip(), title.strip()
else:
    artist, title = "", name
```

`partition` (first occurrence) is intentional — `"Mr. Brightside - Live - Wembley"` splits to `artist="Mr. Brightside"`, `title="Live - Wembley"`. The user can always retype if the split is wrong.

The "what you type is what you get" rule means: if the new value has no `" - "`, artist becomes empty (clearing the prior value). This is predictable; preserving the prior artist invisibly when the user wrote a new label without a dash would be more surprising than helpful.

### Side effect — update lyrics meta

If `cache/<slug>/lyrics/meta.json` exists: load it, replace `artist` and `title`, atomic write back. Other fields (`source`, `lrclib_id`, `album`, `duration_sec`, etc.) are preserved.

If it does **not** exist: create the lyrics cache dir and write a minimal meta with `source="user_rename"`, `lrclib_id=null`, `album=""`, `duration_sec=` from the summary, plus `artist` and `title`. This way a future Refetch from LRCLIB has good seed data.

### Atomicity

Write to `<file>.tmp`, `os.replace`. Single-user app; no cross-process locks needed beyond the OS-level rename atomicity. Same pattern as the existing lyrics save helpers in `webui/webui/lyrics.py`.

### Cache invalidation

`tracks._cache.pop(slug, None)` after the writes. The next `GET /api/tracks/{slug}` will re-read the merged data.

### Response

```json
{
  "display_name": "Charlie Puth - Attention",
  "artist": "Charlie Puth",
  "title": "Attention"
}
```

The client uses this to update the topbar and lyrics tab without an additional GET.

## Server merge — `tracks.get_summary`

`tracks.get_summary(slug)` currently returns the parsed `summary.json`. Wrap with a small read of `user_meta.json` (if present) and merge `display_name` into the `track` dict:

```python
summary = ...  # existing read
um_path = cache_dir / "user_meta.json"
if um_path.is_file():
    user_meta = json.loads(um_path.read_text(encoding="utf-8"))
    dn = (user_meta.get("display_name") or "").strip()
    if dn:
        summary.setdefault("track", {})["display_name"] = dn
return summary
```

All existing API consumers get `summary.track.display_name` for free; no API shape change beyond an additive optional field.

## UI — ✎ pencil + rename modal

### Pencil button

In `topbar.js`, add right of the title pill (outside the picker click target so it doesn't toggle the dropdown):

```js
const editBtn = el("button", {
  class: "title-edit", attrs: { type: "button", title: "Rename track" }, text: "✎",
  onClick: (e) => {
    e.stopPropagation();
    import("./rename-modal.js").then(m => m.showRenameModal({
      slug: window.__currentSlug,
      currentName: summary?.track?.display_name || (summary?.track ? deriveTitle(summary.track.file) : ""),
      onSaved: (resp) => {
        // Update topbar in place
        titleSpan.textContent = resp.display_name;
        document.title = `${resp.display_name} — MusIQ-Lab`;
        // Trigger lyrics tab to refresh its cached meta
        window.__musiqLyricsRefreshMeta?.();
      },
    }));
  },
});
```

Styling: small, low-contrast until hover (matches the existing `.lyrics-refresh` button idiom).

### Rename modal

New file `webui/static/js/ui/rename-modal.js`. Lazy-imported on first click (no startup cost for users who never rename).

Structure mirrors the existing analyze/paste modals: full-screen translucent overlay, a small panel with title, a single text input, hint line, Save / Cancel buttons.

```
┌──────────────────────────────────────────────┐
│ Rename track                                 │
│                                              │
│ ┌──────────────────────────────────────────┐ │
│ │ Charlie Puth - Attention                 │ │
│ └──────────────────────────────────────────┘ │
│ Use "Artist - Title" to populate both fields.│
│                                              │
│                       [ Cancel ]  [  Save  ] │
└──────────────────────────────────────────────┘
```

Behavior:
- Input is autofocused, text pre-selected
- `Enter` saves; `Esc` cancels
- Save button disabled while empty or unchanged
- Save: PATCH; on success call `onSaved(resp)` and close; on 4xx show inline error and keep modal open
- No keyboard-shortcut conflicts: `shortcuts.js` does **not** currently check focus target (verified — no `tagName`/`isContentEditable` checks), so the canvas shortcuts (space, digits) would fire while typing in the modal. Step 3 adds a one-line guard at the top of the shortcuts handler:

  ```js
  if (e.target.matches("input, textarea, [contenteditable=true]")) return;
  ```

  This is the right fix anyway — it also retroactively fixes the lyrics-tab contenteditable that the user flagged as unusable. **This is part of the rename feature, not a separate piece of work.**

### Lyrics tab refresh

`lyrics-tab.js` exposes a tiny `window.__musiqLyricsRefreshMeta` hook that re-fetches the cached lyrics (re-runs `_lazyLoad` essentially) so the header re-renders with the new artist/title. Same cross-tab handoff pattern as the existing `window.__musiqClaudeAsk`.

The lyrics tab's contenteditable header fields stay (the placeholder polish already shipped), but the user is now expected to use the topbar pencil to edit them. We don't strip the contenteditable affordance — it still works for users on a non-canvas page, and ripping it out is a separate decision.

### Surfaces that pick up `display_name`

| Surface | File | Change |
|---|---|---|
| Topbar title pill | `webui/static/js/ui/topbar.js` | `summary.track.display_name || deriveTitle(summary.track.file)` |
| Tools menu label | `webui/static/js/ui/topbar.js` (passed to `showTools`) | Same fallback |
| Track picker | `webui/static/js/ui/track-picker.js` | Same fallback per track in the list |
| Browser `<title>` | `webui/static/index.html` is currently a static `<title>MusIQ-Lab</title>` and never updated. Add a `document.title = ...` set in `main.js` on initial track load AND on rename. | New behavior — minor scope addition |
| Lyrics tab header | `webui/static/js/ui/lyrics-tab.js` | Already reads from lyrics meta — refreshed via the new hook |

## Implementation order

1. **Server first** — `user_meta.json` read/write helpers in a new `webui/webui/user_meta.py` (small module, mirrors `lyrics.py` patterns); `PATCH` route; `tracks.get_summary` merge; `PRESERVE` extension. Tests pass before any UI.
2. **API client + topbar fallback** — wire `renameTrack` into `api.js`, swap topbar/track-picker/main.js to read `display_name` with fallback. No UI change visible yet (no rename button), but every read site is now ready.
3. **Pencil + modal** — `rename-modal.js`, the topbar button, the lyrics refresh hook. Full feature live.
4. **E2E + manual verification** — Playwright spec, browser walkthrough.

Each step is a clean commit boundary.

## Risks and open questions

- **Shortcut focus-guard scope**: the one-line guard in `shortcuts.js` will silence canvas shortcuts not just for the rename modal but also for the lyrics-tab contenteditable, the analyze modal, the paste dialog — everywhere a user types. This is correct behavior across the board. Worth a manual smoke test that nothing relies on the prior (broken) behavior.
- **Server-side trimming**: input is trimmed before validation. We don't normalize Unicode (NFC) — out of scope; the user gets back exactly what they typed.
- **`<title>` is new behavior**: previously the browser tab always read "MusIQ-Lab". Wiring it to track name is a small UX win that comes for free with this feature; doesn't need its own gate.
